## 单次 glow 太快的根因

CSS 时长是 2.5s，但**视觉上的亮带只在中间一小段时间内可见**，所以感觉很短：

```css
@keyframes position-sweep {
  0%   { transform: translate(-100%, -100%); opacity: 0; }
  15%  { opacity: 1; }
  85%  { opacity: 1; }
  100% { transform: translate(100%, 100%); opacity: 0; }
}
```
overlay 容器：`-inset-[40%]`（即 240% × 240%）  
gradient：`linear-gradient(135deg, transparent 42%, gold 50%, transparent 58%)` —— 亮带只有 16% 宽

两个问题叠加：
1. **位移范围过大**：从 `-100%,-100%` 到 `+100%,+100%`，再加上容器本身 240% 尺寸，亮带要横跨大约 4–5 倍卡片宽度，真正经过卡片的时间极短。
2. **亮带过窄**：16% 宽的高斯条带在 2.5s 内扫完这么大距离，肉眼几乎是“闪一下”。

数字 count-up 用的是 ease-out，前半段快后半段慢；而 glow 是匀速大范围位移，因此两者“同时开始、同时结束”但中间节奏完全不同，主观感觉 glow 已经结束，数字还在慢慢爬。

第二次 glow 没问题——它确实由 `onComplete` 触发——但因为第一次太快，用户感觉第一次还没看清就结束了。

## 修复目标
- 第一次 glow 在数字开始跳那一刻启动，在数字到达终值那一刻结束（已实现）。
- 第二次 glow 紧接第一次开始，时长完全相同。
- **单次 glow 视觉上要充满整个 2.5s**，亮带在卡片可见区域里慢慢扫过，而不是“嗖一下”。

## 修复方案

### 1. 缩小位移范围
将 keyframe 的位移从 `-100% → +100%` 改为大约 `-60% → +60%`，让亮带主要时间都在卡片可见区域内，而不是在 overlay 容器的边缘外。

### 2. 加宽亮带
将 gradient 的可见宽度从 16% 扩到约 35–45%：
```css
linear-gradient(135deg, transparent 30%, gold 50%, transparent 70%)
```
这样即使匀速扫，可见时间也会拉长。

### 3. 收窄 overlay 容器
把 `-inset-[40%]` 改为 `-inset-[15%]` 或 `-inset-[20%]`，减少容器的“空走”面积。

### 4. 调整 opacity 曲线
让亮带在 2.5s 内大部分时间都是高不透明度：
```css
0%   opacity: 0
8%   opacity: 1
92%  opacity: 1
100% opacity: 0
```

### 5. 与 count-up 节奏匹配（可选）
将 keyframe 用 `cubic-bezier(0.33, 1, 0.68, 1)`（ease-out）替代当前的 `cubic-bezier(0.22, 0.61, 0.36, 1)`，让亮带和数字一样前快后慢，结束时刻视觉上更同步。

### 6. 不动 JS 逻辑
触发链路（数字第一帧 > 0 启动第一次 glow，count-up 完成时启动第二次）已经正确，不再改动。

### 预期结果
- 第一次 glow 全程 2.5s，亮带从卡片左上慢慢扫到右下，结束时数字刚好到达终值。
- 第二次 glow 立刻接上，同样 2.5s，节奏一致。
- 不再出现“一闪而过”的感觉。
